## Critical compliance findings

### C3.1 [BLOCKING for commercial launch] — Yahoo Finance commercial redistribution

**Source:** R4 F4.5, R7 F7.8 (cross-confirmed).

**Evidence:**
- yahoo-finance2 is the primary feed in `app/api/prices/route.ts:36`, `app/api/chart/[ticker]/route.ts`, `app/api/options/[ticker]/route.ts`, `app/api/fundamentals/[ticker]/route.ts`, `app/api/stream/[ticker]/route.ts`, `app/api/crypto/btc/route.ts` (likely), and several backtest scripts.
- Yahoo Finance ToS §2 restricts use to "personal, non-commercial."
- yahoo-finance2's own README disclaimer recommends migration to Polygon/Refinitiv for production use.
- Phase 13 platform target: institutional users / paying researchers — explicit commercial use case.

**Citation:**
- Yahoo Finance Terms of Service: https://policies.yahoo.com/us/en/yahoo/terms/utos/index.htm (§2: "personal, non-commercial").
- yahoo-finance2 npm README disclaimer.

**Verdict:** **BLOCKING.** I3 will NOT sign off on any commercial deployment until one of:
1. Migration to a licensed feed (Polygon, Refinitiv, IEX Cloud) — preferred.
2. Written legal opinion from securities-data counsel approving Yahoo use under specific facts.
3. Explicit non-commercial deployment scope (research-only, no paying users).

**S4 deliverable:** `docs/DATA_SOURCE_MIGRATION.md` documenting the Polygon migration path AND `components/ComplianceBanner.tsx` rendered on every page (already exists per Phase 12 inventory; verify it's wired and texts the right thing).

### C3.2 [HIGH] — Bloomberg bridge license — verify scope

**Source:** R4 deferred; `lib/data/bloomberg/bridgeClient.ts` exists.

**Evidence:** The bridge connects to a Bloomberg Terminal or BLPAPI license. Bloomberg licensing is per-seat and prohibits redistribution. If multiple QUANTAN users receive Bloomberg-sourced data through one bridge license, this is a TOS violation.

**Citation:** Bloomberg L.P. Subscription Agreement standard terms (per-user redistribution restrictions).

**Verdict:** Not BLOCKING for non-commercial / personal use, but BLOCKING for any deployment serving > 1 user with Bloomberg data. Document the deployment topology in `docs/architecture.md` and include a per-user license verification step.

**S4 deliverable:** explicit license-scope documentation; toggle Bloomberg bridge OFF for any multi-user deployment unless per-user license confirmed.

---

### C3.3 [HIGH] — Options data redistribution (US OPRA)

**Source:** R3 deferred; options API routes consume yahoo-finance2's options endpoint.

**Evidence:** US options data flows from OPRA (Options Price Reporting Authority). Real-time OPRA data redistribution requires OPRA license fees. yahoo-finance2 returns 15-minute-delayed quotes (typical for free Yahoo tier), which avoids real-time OPRA fees. Confirm the platform clearly labels options data as "delayed" OR migrates to a licensed feed.

**Citation:** OPRA Rules of the Plan (SEC-approved); SIFMA guidance on market data redistribution.

**Verdict:** Add explicit "DELAYED 15 MIN" label to all options components and `/api/options/[ticker]` response payload. If real-time options data is ever required, license OPRA via Polygon.

**S4 deliverable:** delayed-data label rendered in `components/options/*` and in the API response timestamp metadata.

### C3.4 [MEDIUM] — Crypto data sources — verify

**Source:** `app/api/crypto/btc/*` — head of `app/api/crypto/btc/route.ts` shows it parses interval/limit; underlying source not yet read.

**Verdict:** Document the source in `lib/crypto.ts` and `docs/DATA_SOURCE_MIGRATION.md`. Add attribution to UI if required by source ToS.
