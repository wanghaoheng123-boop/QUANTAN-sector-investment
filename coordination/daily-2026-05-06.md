## Critical decisions today

### D6 — F1.7's reframing forces the institutional-trust narrative

The original day-0 framing of F1.7 was "no correlation-adjusted Kelly" — a missing feature. The day-1 read shows it is **falsely advertised**: header docstring claims it, config exposes `correlationGate: 0.20`, state struct `tickerDailyReturns` is initialized — and **none of it is wired**. For institutional-grade software this crosses from "missing capability" into "false claims about risk management."

**Decision:** S2 day-1 PR must implement correlation-adjusted Kelly OR remove the documentation/config until implemented. **No silent gap.**

### D7 — Sortino is now a SSOT-week priority

R2 F2.1 + I1 spot-check #2 + R1 F1.10 already converged on engine.ts vs indicators.ts. Day-1 read found a THIRD variant in portfolioBacktest.ts. **Decision:** Day-1 of S2 starts with a single PR consolidating all three Sortino implementations into `lib/quant/indicators.ts:sortinoRatio()` with:
- canonical denominator: `n_d` (negative-returns count)
- canonical MAR: configurable, default 0 with documented rationale
- minimum sample: `n_d ≥ 30` (per F1.9)
- numerator: excess return `(mean - rfDaily)` matching MAR

Three call-sites import the canonical version; equivalence test asserts byte-equal output across the three.

### D8 — F3.9 (chain.ts hardcoded RFR) feeds the FRED scaffold

R3's F3.9 (RFR=5.25% in chain.ts) plus R1's F1.4 (RFR=4% in engine.ts) forces a single FRED-backed `getRiskFreeRate()` helper to land in S2 — **not** S3 as originally scoped. Two SSOT violations of the same constant in the same codebase is enough to promote the data scaffold.

### D9 — I1 must re-spot-check F1.7 and F1.16

The day-1 deltas materially change S2's scope. **Decision:** I1 produces `inspections/I1-S1-day1.md` re-verifying:
